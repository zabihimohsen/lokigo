# lokigo Growth Plan (8 Weeks)

## Main purpose

Become known as the Go developer who solved Loki logging for sidecar-less platforms (especially Railway), then carry that credibility into the next project.

## Positioning statement

`lokigo` is a focused, production-ready Loki client for sidecar-less Go deployments with explicit reliability controls and low dependency surface.

## Execution plan

### Week 1 — Trust and packaging

- Make releases intentional (manual semver tags, clean release notes).
- Keep changelog synced with actual tags.
- Add contribution/security docs and issue/PR templates.
- Enable GitHub Discussions and set repository topics.

### Weeks 2-3 — Proof

- Publish a public example app: Go API + Railway + lokigo + Grafana Cloud.
- Add copy/paste production config snippets to README/docs.
- Publish benchmark notes with tradeoffs (payload size vs debuggability).
- Cut `v0.2.0` with clear migration notes.

### Weeks 4-6 — Distribution

- Publish one deep technical post about sidecar-less Loki on Railway.
- Cross-post to Dev.to, Hashnode, and Grafana community forum.
- Submit to awesome-go (Logging section).
- Post short proof-focused updates with real screenshots and numbers.

### Weeks 7-8 — Transition criteria

Start next project when at least two criteria are met:

- 30+ GitHub stars
- 3+ external issues or PRs
- 2+ production testimonials

Then position the new repo as "from the author of lokigo".

## Weekly KPI checklist

- stars (net new)
- forks
- external contributors
- docs/example traffic
- production users willing to provide testimonial
